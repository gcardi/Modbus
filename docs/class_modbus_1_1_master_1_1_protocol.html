Detailed Description

Abstract base class for all Modbus master transport implementations.

Protocol implements the Non-Virtual Interface (NVI) pattern to decouple the public API from transport-specific implementations. Concrete subclasses supply the transport logic (TCP, UDP, RTU, or Dummy) by implementing the Do…() virtual methods.

NVI Pattern Overview:

  • All public methods (Open, Close, ReadCoilStatus, ReadInputStatus, ReadHoldingRegisters, ReadInputRegisters, ForceSingleCoil, PresetSingleRegister, ForceMultipleCoils, PresetMultipleRegisters, MaskWrite4XRegister, ReadWrite4XRegisters, etc.) are concrete and non-virtual; they perform input validation and orchestration.
  • Each public method forwards to a corresponding protected Do…() virtual method that performs the actual transport-specific work (e.g., DoOpen(), DoReadHoldingRegisters()).
  • Subclasses override only the Do…() virtual methods; they never override the public API.
  • This ensures consistent contract enforcement and logging/validation across all transports.

Virtual Method Naming Convention:

  • Virtual hooks use the "Do" prefix: DoOpen, DoClose, DoReadHoldingRegisters, etc.
  • These methods are declared in the protected section and marked virtual.
  • Subclasses use override keyword when implementing these hooks.

Idempotency:

  • Open() and Close() are idempotent: calling Open() when already connected, or Close() when already disconnected, is a safe no-op.

◆ DoReadCoilStatus()

void Modbus::Master::Protocol::DoReadCoilStatus ( Context const & Context,
CoilAddrType StartAddr,
CoilCountType PointCount,
CoilDataType * Data )
protectedpure virtual

Pure virtual hook for reading holding registers (FC03).

This is the NVI pattern implementation hook. Subclasses override this method to provide transport-specific logic for FC03 read operations. The public API method ReadHoldingRegisters() calls this; subclasses must not override ReadHoldingRegisters().

Subclasses must implement this method to:

  1. Construct a Modbus request PDU (function code 03, start address, point count).
  2. Send it via the transport (TCP, UDP, RTU, or Dummy).
  3. Receive and parse the response, validating CRC/MBAP headers.
  4. Extract the register values and populate the Data buffer.

Implemented in Modbus::Master::DummyProtocol, Modbus::Master::RTUProtocol, and Modbus::Master::TCPIPProtocol.

FIFOCountType Modbus::Master::Protocol::ReadFIFOQueue ( Context const & Context,
FIFOAddrType FIFOAddr,
RegDataType * Data )
inlinenodiscard

Reads the contents of a FIFO queue of registers from the slave (FC24).

Parameters
ContextTransaction context (slave address, transaction ID).
FIFOAddrAddress of the FIFO pointer register on the slave.
[out]DataPointer to output buffer; must hold at least 31 elements (the Modbus-defined maximum FIFO depth).
Returns
The number of register values actually read (0–31).

The slave returns the current FIFO count followed by that many 16-bit register values. If the FIFO is empty the return value is 0 and Data is untouched.

Exceptions
EIllegalDataAddressif the FIFO pointer address is invalid on the slave.
EIllegalDataValueif the FIFO contains more than 31 values.
EIllegalFunctionif the slave does not support FC24.
EBaseExceptionon communication error or timeout.
